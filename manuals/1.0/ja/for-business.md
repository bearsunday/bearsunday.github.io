# なぜBEAR.Sundayなのか

PHPフレームワークを評価する技術リーダーと意思決定者のためのガイド。

---

## エグゼクティブサマリー

フレームワーク選定の最大のリスクは、ビジネスの寿命とフレームワークの寿命の不一致です。BEAR.Sundayは2015年以降破壊的変更がなく、PHP5時代のコードがPHP8.5で動作します。ビジネスロジックはフレームワークに依存しない純粋なPHPで書かれるため、将来フレームワークを離れる際も資産として再利用できます。

## フレームワーク選定の最大のリスク

Webフレームワーク選定における最大のリスクは、機能の不足やパフォーマンスの問題ではありません。最大のリスクは、**「ビジネスの寿命」と「フレームワークの寿命」の不一致**です。

具体的には、ビジネスロジックがフレームワークに深く依存（密結合）し、フレームワークの衰退や破壊的変更に追従できなくなることが最も致命的なリスクとなります。

### ライフサイクル・ミスマッチの罠

多くのプロダクトは5年、10年と稼働し続けます。しかしフレームワークの「流行のサイクル」は3〜5年。メジャーバージョンアップで破壊的変更が行われます。

なんらかの原因で特定バージョンへの過剰適応が進みすぎ、かつ進化に追従できなくなったとき、セキュリティリスクを抱えたまま塩漬け運用せざるを得なくなってしまいます。

### 選定時の最大の問い

したがって、選定時の最大の問いは、「便利か？」ではなく、

> **「将来、このフレームワークを捨てるときにどれだけ苦労するか？」**

であるべきです。

## BEAR.Sundayの回答

BEAR.Sundayの設計は、この問いに正面から答えることができます。

### リスクへの対処

| リスク | BEAR.Sundayの対処 |
|--------|-------------------|
| **ライフサイクル・ミスマッチ** | 2015年以降、破壊的変更なし。「バージョンアップ」という概念自体がない |
| **密結合による塩漬け** | 強制マイグレーションゼロ。PHP5時代のコードがPHP8.5で動作 |
| **フレームワーク＝アプリケーション化** | アプリ合成により段階的移行が可能。他FWから呼び出し可能。全面書き直し不要 |

### BEAR.Sundayだけの強み

| 強み | 効果 |
|------|------|
| **CDN中心設計** | オリジンダウンでもサービス継続。インフラコスト削減 |
| **マイクロサービスなしのアプリ合成** | 複数アプリを同一プロセスで連携。HTTP不要 |
| **脆弱性発見が容易なアーキテクチャ** | 明示的DI、マジックなし、SQLファイル分離。監査しやすい |
| **最小ブートストラップコスト** | コンパイル時DI解決。最小のフレームワークオーバーヘッド |
| **SQLの予防的品質管理** | デプロイ前にスロークエリを検出。治療ではなく予防 |
| **情報設計（ALPS）からの開発** | ビジネスと技術の共通言語。技術変更に耐える |
| **自動生成されるAPIドキュメント** | コードとドキュメントが常に同期。陳腐化しない |

## 総所有コスト

### 1年目：同程度の投資

初期開発コストは他のフレームワークと同等です。BEAR.Sundayの学習曲線は以下で相殺されます：

- 最初からクリーンなアーキテクチャ
- 「マジック」動作が少ない
- 明示的な依存関係

ただし、DI/RESTの基礎知識は前提となります。これらの原則を理解している開発者にとっては、フレームワーク固有の流儀を学ぶ必要がなく、むしろ習得が早いです。

### 2-5年目：差が現れる

| 活動 | 一般的なフレームワーク | BEAR.Sunday |
|------|------------------------|-------------|
| セキュリティパッチ | 適用後に破壊がないかテスト | そのまま適用 |
| PHPアップグレード | フレームワーク更新が必要な場合も | シームレス |
| 機能追加 | 既存アーキテクチャとの戦い | クリーンに拡張 |
| 新メンバー | フレームワークの癖を学習 | 原則を学習 |

### 5-10年目：複利効果

- 大規模マイグレーションプロジェクトなし
- オリジナルのコードベースがメンテナンス可能なまま
- モジュール更新によるパフォーマンス改善（例：CDN中心、並列実行）
- 何年も前に書かれたコードが修正なしで新機能を獲得

## CDN中心アーキテクチャ

### 分散キャッシュフレームワーク

BEAR.Sundayは**サーバーサイドキャッシュ**（Redis、APCu）、**共有キャッシュ**（CDN）、**クライアントサイドキャッシュ**（ブラウザ）を統合した分散キャッシュフレームワークを提供します。

従来、アプリケーションロジックを必要とするコンテンツは「動的」でありCDNでキャッシュできないと考えられてきました。しかし、BEAR.Sundayのリソース指向設計は、イベント駆動のキャッシュ無効化を実現します：

- TTLベースではなく、リソース変更時に関連キャッシュを即時無効化
- オリジンサーバーの負荷を大幅に軽減
- スケール時のインフラコスト削減

### 障害時の継続稼働

**オリジンサーバーがダウンしても、CDNがキャッシュからコンテンツを配信し続けます。** ユーザーはサービス停止に気づきません。これは従来のアーキテクチャでは実現困難な耐障害性です。

### コスト削減効果

CDN中心設計により：

- サーバー台数の削減
- トラフィック急増時も追加インフラ不要
- 障害対応の複雑なフェイルオーバー構成が不要

## マイクロサービスなしのアプリケーション合成

複数のアプリケーションを連携させる必要がある場合、通常はマイクロサービスアーキテクチャを検討します。しかし、マイクロサービスには課題があり多くの組織が失敗してきました：

| 課題 | マイクロサービス | BEAR.Sunday Import |
|------|------------------|---------------------|
| ネットワークオーバーヘッド | HTTPリクエスト必須 | 同一プロセス内で直接呼び出し |
| インフラ複雑性 | サービスごとにデプロイ | Composerパッケージとして統合 |
| データ整合性 | 分散トランザクション問題 | 同一プロセスで解決可能 |
| デバッグ | 複数サービスをトレース | 単一アプリとしてデバッグ |

複数のBEAR.Sundayアプリケーションを、HTTP APIなしに単一システムとして連携できます。マイクロサービスの利点（明確な境界、独立したデプロイ）を維持しながら、モノリスの利点（シンプルさ、パフォーマンス）も得られます。

## セキュリティ：脆弱性発見が容易なアーキテクチャ

### 監査しやすい設計

BEAR.Sundayのセキュリティ上の強みは「見通しの良さ」です：

**内側からの監視：**
- DIによる明示的な依存関係 — 何がどこに注入されるか追跡可能
- マジックメソッドやグローバル状態なし — コードパスが予測可能
- AOPによる横断的関心事の分離 — 認証・認可ロジックが一箇所に集約

**外側からの監視：**
- 静的解析ツール（PHPStan、Psalm）との高い親和性
- 型安全な設計 — 実行前に多くの問題を検出
- SQLが独立ファイル — SQLインジェクションの監査が容易

### 攻撃対象になりにくい

セキュリティの観点で見落とされがちな事実：「人気フレームワーク」以外は攻撃対象になりにくいこと。

攻撃者は投資対効果を考えます。LaravelやWordPressを狙えば何百万ものサイトに同じ攻撃が有効ですが、BEAR.Sundayを狙っても得られる成果は限定的です。これは副次的な効果ですが、実際のリスク軽減要因です。

---

## パフォーマンス：最小ブートストラップコスト

### 限界まで最適化されたアーキテクチャ

- **コンパイル時DI解決** — 他のDIコンテナは実行時に依存解決。BEAR.Sundayは事前に解決済み
- **ルートオブジェクトの再利用** — アプリケーション全体が単一の変数。リクエストをまたいで再利用
- **イミュータブル設計** — 状態を持たないため、Swoole等の非同期サーバーでも安全に動作

フレームワークを使う「パフォーマンスコスト」がほぼありません。

### SQL品質の予防的管理

従来のアプローチは「遅くなってから対処」です。AWSのログでスロークエリーを見つけ、本番で問題が起きてから修正する—これは治療です。

BEAR.Sundayエコシステムでは、デプロイ前に品質の悪いSQLを検出します—これは予防です：

- フルテーブルスキャン、非効率なJOINを自動検出
- AI支援による最適化提案
- 最適化前後のコスト比較レポートが自動生成

**なぜこれが可能なのか**: BEAR.SundayではSQLは独立したファイルで管理され単独での実行が容易です。SQLがコード内に埋め込まれたり、生成されず独立しているため、自動的な分析・改善が容易です。

## 情報設計（ALPS）からの開発

多くのプロジェクトは「どんなAPIエンドポイントが必要か」から始まります。BEAR.Sundayでは、より本質的な「情報設計」から始めることができます。

### ALPSによる情報設計

ALPS（Application-Level Profile Semantics）はアプリケーションの本質を定義します：

- **語彙** — 「ユーザー」「注文」などビジネス用語の定義
- **構造** — 情報の分類と関係性
- **操作** — 何ができるか、どう遷移するか

これにより、要件定義フェーズからビジネスチームと技術チームが同じドキュメントを共有できます。

### 技術変化への耐性

ALPSで定義されたビジネスドメインは、実装技術から独立しています。REST → GraphQL への移行、モノリス → マイクロサービス への変更、フレームワークの刷新—技術が変わっても、ドメインモデルは継続して活用できます。

## 自動生成されるAPIドキュメント

### コードとドキュメントの同期

| 形式 | 用途 |
|------|------|
| OpenAPI 3.1 | フロントエンドがAPIクライアントを自動生成 |
| JSON Schema | リクエスト/レスポンスの型定義を共有 |
| HTML ApiDoc | 開発者向けリファレンス |

ドキュメントはコードから自動生成されるため：

- 「ドキュメントが古い」問題が発生しません
- APIの変更が即座にドキュメントに反映されます
- フロントエンドは常に最新の型定義を取得可能です

「APIの仕様を確認したい」「型定義をください」というやり取りが不要になります。コミュニケーションリスクを低減します。

## よくある懸念への回答

### 「有名ではない」

人気はビジネス要件ではありません。以下を考慮してください：

- **採用**: BEAR.Sundayは標準的なPHP、PSR標準、よく知られたパターン（DI、REST）を使用。熟練したPHP開発者なら誰でも貢献できます。
- **コミュニティ**: 小さいが集中している。メンテナーへの直接アクセス。
- **継続性**: 10年以上の一貫した開発、破壊的変更なし。

### 「開発者は見つかるのか」「オンボーディングに時間がかかるのでは」

BEAR.Sundayはすべてが明示的で宣言的です：

- 依存関係はコンストラクタに宣言
- バリデーションはJSON Schemaで宣言
- SQLは独立したファイルに記述
- リソースURIがそのままルーティング

境界がはっきりしていて、その入出力が宣言的です。

### 「フレームワークに合わせるのが大変では」

BEAR.Sundayに「過剰適応」する必要はありません。アプリケーションの本質はドメインロジックとSQLです。BEAR.Sundayのリソースは、それらに薄いUIレイヤーを被せたもの。フレームワーク固有の書き方にコードを歪める必要がなく、ドメインの純粋さを保てます。

## もしBEAR.Sundayをやめることになっても

フレームワーク選定で最も恐れられるのは「ロックイン」です。BEAR.Sundayは、この懸念に正面から答えます。

### 成果物が再利用可能

| 成果物 | 再利用性 |
|--------|----------|
| SQLファイル | 標準SQL。どのフレームワークでも使用可能 |
| クエリインターフェース | 純粋なPHPインターフェース。実装を差し替え可能 |
| リソースクラス | 純粋なPHPクラス。他のフレームワークから呼び出し可能 |
| ALPS/JSON Schema | 標準仕様。言語・フレームワーク非依存 |
| Ray.Di/Ray.Aop | 独立パッケージ。Laravel/Symfonyでも使用可能 |
| ドメインロジック | フレームワークに依存しない設計 |

BEAR.Sundayを離れても多くの部品が再利用可能です。

### 外部からのリソース呼び出し

LaravelやSymfonyなど他のフレームワークから、BEAR.Sundayリソースをそのまま呼び出せます：

```php
$resource = Injector::getInstance('MyVendor\MyApp', 'prod-app', $appDir)
    ->getInstance(ResourceInterface::class);
$user = $resource->get('/user', ['id' => 1]);
```

移行期間中も既存リソースを活用しながら、新システムを構築できます。

### 学習は投資になる

BEAR.Sundayで学ぶのはフレームワーク固有の流儀ではなく、普遍的なソフトウェア原則です：

- 依存性注入（DI）— あらゆる言語・フレームワークで通用
- RESTアーキテクチャ — Web開発の基盤知識
- クリーンアーキテクチャ — 設計原則

フレームワーク固有の知識は、そのフレームワークを離れれば価値を失います。BEAR.Sundayで得た知識は、どこでも活かせる資産になります。
