Injected Dependency Framework

In BEAR.Sunday DI is not just a bolt on or handy tool. Everything you code in your application is coded against an interface or abstract class, this means that you are never creating a hard dependency on libraries or components that might change in the future. more... This means you can upgrade with new libraries without changing your core application whatsoever.

Aspect Wrapped Framework

In BEAR.Sunday we recognize that crucial per request flexibility is key to the business of web applications. With AOP included in BEAR testable aspects that are not part of your business logic can wrap your core concerns without them even knowing about it. more... This means you can write clean domain logic without cluttering it with the likes of caching, logging and authentication code. These will instead be applied elegantly.

Why you’ll like BEAR.Sunday

Real flexibility and best practicies with performance.

No own components. Chosen from the best.

BEAR.Sunday offers no libraries of its own. PHP namespaces, PSR, a new coding github culture, unit testing, a new trend of library oriented frameworks... these have all happened recently and push us to a more library oriented way of thinking. So BEAR.Sunday chooses not to have it’s own components, it uses others from Aura, Symfony and Zend framework etc.

Everything is a resource. API is core, REST is centered.

In BEAR.Sunday everything is a REST resource which leads to far simpler design and extensibility. Interactions with your database, services and even pages and sections of your app all sit comfortably in a resource which can be consumed or rendered at will.

Decouple everything. Code intention, then bind implementation.

DI removes coupling between application objects, while AOP decouples application concerns. BEAR.Sunday is also abstraction framework. DSL, Annotation, URI, Interface, Aspects and Hypermedia. These abstraction technology encourage intentional-oriented code.

Very Fast. DRY Runtime boosts performance.

It has a unique Runtime DRY*. Don’t repeat the same procedure again and again. the whole application object graph can be serialized and stored in memory. It is not recreated on each request. All dependency are injected and initialized..

Very Testable. Because everything is injected.

Test friendly architecture. Stateless resource request, full dependency injection, and the separation of value and view make test easy.
